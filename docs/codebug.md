# codebug — co-execution debugger

`tools/codebug.mjs` runs **calcite** and the **JS 8086 reference emulator**
(`tools/js8086.js`) in lockstep against the same CSS program, exposing a
unified HTTP API so you can step, inspect, send keyboard input, and diff
both sides to find the exact tick where they diverge.

Two reasons this tool exists:

1. **Debugging DOS programs that misbehave in calcite.** When Rogue crashes
   on enemy contact or Snake won't boot, the interesting question is
   "where does calcite's execution first differ from the reference?"
   — not "what does calcite show on its own." codebug answers that.
2. **Catching CSS-DOS transpiler bugs.** Most bugs are in the generated
   CSS, not in calcite. codebug surfaces them as divergences between the
   two interpreters of that same CSS.

The JS emulator is the ground truth. If calcite and the reference disagree
at tick N, either the CSS is wrong (most likely) or calcite is wrong.

## Modes

Only **DOS mode** is supported. The JS side loads the Gossamer-DOS BIOS at
`F000:0000`, the DR-DOS kernel at `0060:0000`, and the FAT12 disk image at
`D000:0000`, then starts execution at `F000:bios_init`. The calcite side
runs whatever CSS was generated by `../CSS-DOS/transpiler/generate-dos.mjs`,
which sets up exactly the same memory layout. At tick 0 both sides have
identical registers.

If you need to debug non-DOS programs, use the existing `tools/compare.mjs`
or `tools/fulldiff.mjs` tools.

## Quick start

```sh
# 1. Generate CSS for your DOS program (if not already done)
node tools/calc-mem.mjs programs/rogue.com              # prints memory size
node --max-old-space-size=8192 \
  ../CSS-DOS/transpiler/generate-dos.mjs programs/rogue.com \
  -o output/rogue.css --mem 0x4c000

# 2. Build calcite-debugger
cargo build --release -p calcite-debugger

# 3. Start codebug
node tools/codebug.mjs output/rogue.css

# codebug listens on :3334 and drives calcite-debugger on :3333 as a child.
# Use --port=N and --calcite-port=N to change those.
```

In another shell:

```sh
# Confirm both sides booted with identical state
curl localhost:3334/info
curl localhost:3334/regs                    # all diffs should be []

# Step 1000 ticks on both sides
curl -X POST localhost:3334/step -d '{"count":1000}'

# Check if they still agree
curl localhost:3334/regs

# Render both screens side-by-side
curl localhost:3334/screen

# Queue a key (ASCII 'j', scancode 0x24)
curl -X POST localhost:3334/key -d '{"ascii":106,"scancode":36}'
# Next /step call will flush it to both sides.

# Compare registers + specific memory ranges
curl -X POST localhost:3334/compare \
  -d '{"memory":[{"addr":0,"len":1024},{"addr":753664,"len":4000}]}'
# Memory range examples:
#   0x000-0x400  = IVT (1024 bytes)
#   0xB8000      = VGA text buffer (4000 bytes)

# Shut down (stops calcite-debugger and exits)
curl -X POST localhost:3334/shutdown
```

## Endpoints

| Method | Path | Body | Description |
|--------|------|------|-------------|
| GET  | `/info`    | — | Current ticks for both sides, agreement flag, calcite session info, key queue depth. |
| POST | `/step`    | `{count}` | Advance both sides by `count` ticks. If a key is queued, it is written to `0x500` on both sides **before** the batch runs. Returns `{ticks_js, ticks_calcite, key_flushed, js_error}`. |
| POST | `/key`     | `{ascii, scancode}` or `{value}` | Queue a key for next step. `value = (scancode << 8) \| ascii`. Queue is FIFO and shared by both sides. |
| GET  | `/regs`    | — | Both sides' registers plus a `diffs` array listing any mismatched registers. |
| GET  | `/screen`  | — | Both VGA buffers as text, a `side_by_side` diff string, and an `agrees` bool. Both sides are normalized to printable ASCII so CP437 vs ASCII cosmetic differences don't show up. |
| POST | `/compare` | `{memory?}` | Always returns register diffs. If `memory` = `[{addr,len}, ...]` is given, also diffs those byte ranges and returns `memory_diffs[].diffs[]` with each mismatching `{addr, js, calcite}`. |
| POST | `/seek`    | `{tick}` | Reset both sides and run forward to `tick`. Calcite uses its own auto-checkpoints so this can be fast. The JS side has no checkpoints in phase 1 — every backwards seek is a full replay from tick 0. Use sparingly. |
| POST | `/shutdown` | — | Stop both sides and exit. |

## How keyboard input works

Both sides' BIOS (`CSS-DOS/bios/gossamer-dos.asm`, `int16h_handler`) polls
linear address `0x500` for keystrokes. The word is stored as
`(scancode << 8) | ascii`. The `INT 16h` handler busy-waits until the word
is nonzero, reads it, then writes `0` back to consume it.

Because both sides use the exact same BIOS and the exact same polling
address, codebug injects keys **symmetrically**:

- calcite: `POST /key {value}` to calcite-debugger, which sets
  `state.keyboard` (which is mirrored into `read_mem(0x500)`).
- JS ref: write two bytes into `jsMemory[0x500..0x502]`.

One keystroke is injected per `/step` call — this matches how real polling
would behave (one keypress per poll window). If you queue several keys in
a row, each subsequent `/step` flushes one.

**Always send key input via codebug, not directly to calcite-debugger.**
Sending to one side only will immediately desync the two emulators.

## Expected workflow for debugging a divergence

```sh
# 1. Confirm they match at tick 0
curl localhost:3334/regs
# → diffs: []

# 2. Binary-search for the first tick where they disagree.
#    (Phase 2 will automate this — for now it's manual.)
for N in 1000 10000 50000 100000; do
  curl -sX POST localhost:3334/step -d '{"count":'$N'}' > /dev/null
  D=$(curl -s localhost:3334/regs | jq '.diffs | length')
  T=$(curl -s localhost:3334/regs | jq '.tick_js')
  echo "tick $T: $D register diffs"
done

# 3. Once you know roughly when it breaks, /seek and step in smaller
#    batches to zoom in:
curl -X POST localhost:3334/seek -d '{"tick":95000}'
curl -X POST localhost:3334/step -d '{"count":100}'
curl localhost:3334/regs

# 4. When the first diverging tick is found, inspect:
#    - Register diffs (what differs)
#    - /screen (what the program is displaying)
#    - /compare with a memory range covering the likely code area
#
# The divergence is usually one of:
#    a) Calcite evaluated a CSS expression wrong → fix calcite.
#    b) The CSS emits the wrong expression → fix CSS-DOS transpiler
#       (../CSS-DOS/transpiler/src/).
#    c) An INT handler in gossamer-dos.asm has a bug.
```

## Architecture

```
+-------------------+           +-----------------------+
|  tools/codebug.mjs|           | calcite-debugger      |
|  (Node process)   |  HTTP     | (child process,       |
|                   |<--------->|  :3333 by default)    |
|  - js8086 runs    |           |                       |
|    in-process     |           |  drives calcite-core  |
|  - HTTP server    |           |  Evaluator + State    |
|    :3334          |           +-----------------------+
+-------------------+                        ^
         ^                                   |
         | HTTP                              |
         |                                   |
      curl / your script
```

- `codebug.mjs` is a Node script that spawns `calcite-debugger` as a child,
  polls `/info` until it's ready, then starts its own HTTP server.
- The JS 8086 emulator runs in-process in the same Node script (no extra
  process, no extra RTT for per-tick work).
- Each `/step` call: codebug makes one HTTP call to calcite (batched
  `POST /tick`) and loops `jsCpu.step()` locally. Keyboard injection
  happens once per batch, not per tick.
- Divergences are computed in Node because the JS side's register format
  is already there and it's cheaper than shipping both states to Rust.

## Known limitations (to be addressed in phase 2)

- **No `run-until-diverge`.** You have to step and check manually. Phase 2
  will add an endpoint that samples every N ticks, finds the first
  mismatching sample, then bisects to pinpoint the exact tick.
- **No JS-side snapshots.** Backwards seeks on the JS side are a full
  replay from tick 0. For a program at tick 1,000,000 this is slow. Phase 2
  will add periodic snapshots of `jsMemory` and `cpu.getRegs()` mirroring
  calcite's auto-checkpoints.
- **CP437 vs ASCII screen text.** Calcite renders CP437 glyphs for bytes
  outside 0x20-0x7E, JS renders spaces. Screen diffs normalize both sides
  to ASCII to avoid cosmetic false positives, but this means some box-
  drawing content can't be visually compared. Byte-level `/compare` with
  the VGA range is the authoritative check.
- **INT tracing is only in `tools/ref-dos.mjs`.** codebug doesn't
  reproduce that feature yet. If you need INT logs, run `ref-dos.mjs`
  alongside.
