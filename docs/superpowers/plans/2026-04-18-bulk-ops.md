## Background engineers need to know

### The problem and why the old approach failed

Calcite ticks the whole CPU once per emitted byte. A 64 KiB REP STOSB takes ~64 000 ticks. The previous spec (`2026-04-18-memory-fill-op-design.md`, now superseded) tried a compile-time peephole to recognise a byte-fill loop in calcite's bytecode. The diagnostic probe (`probe-bytecode-shape`, commit `badb0c7`) proved this cannot work: CSS-DOS compiled bytecode contains **zero `StoreMem` ops**. All memory writes go through broadcast-write ports that emit one byte per tick. The "loop" is calcite's tick scheduler, not CSS code. No static pattern exists to match.

### Why the new approach does work

Chrome evaluates every memory-cell property assignment in parallel per tick. A 64 KiB fill could complete in ONE Chrome tick if every memory cell's assignment includes an "am I in the bulk-op range?" clause. This is real Chrome work (~1 M range checks per active tick). Calcite recognises the range-predicate pattern across all cells and collapses it to one native `memory.fill` call.

**Rule compliance** (per `CLAUDE.md` commit `c62b76a`):
- Every memory cell does honest Chrome work (range check every tick bulk op is active).
- No dummies. No side-channels. `--bulkOpKind` is a first-class state property consumed by every cell.
- Chrome remains authoritative; calcite just pattern-matches.

### CSS-DOS architecture pointers (V4)

- BIOS assembly lives in `bios/css-emu-bios.asm`. DOS interrupt handlers are in `bios/handlers.asm`. Splash/boot C code in `bios/splash.c` (OpenWatcom C, 8086 real mode).
- Transpiler entry: `transpiler/generate-dos.mjs`. Pattern tables: `transpiler/src/patterns/*.mjs` (alu, control, mov, etc.). Memory-cell CSS emission: `transpiler/src/memory.mjs`. Top-level CSS emitter: `transpiler/src/emit-css.mjs`.
- Cardinal rule still holds: Chrome is authoritative. If Chrome disagrees with calcite, calcite is wrong.
- V4 = every instruction completes in one CSS tick. That invariant must hold for INT 2F/FE too.

### Calcite architecture pointers

- Op enum in `crates/calcite-core/src/compile.rs` (`pub enum Op`, ~line 43–315 after PR 1 commits).
- Five places every new Op must be handled: `op_slots_read`, `op_dst`, `map_op_slots`, `seed_from_parent`, and the three execution functions (`exec_ops`, `exec_ops_profiled`, `exec_ops_traced`). All in `compile.rs`.
- Pattern recognition at the Assignment/CSS layer: `crates/calcite-core/src/pattern/broadcast_write.rs` — the closest analogue. New detector `memory_fill.rs` will live alongside it.
- Diagnostic probe to use during debugging: `cargo run --release --bin probe-bytecode-shape` — dumps op-kind histogram and near-miss patterns from a compiled CSS file.
- Correctness gate for the splash workload: `cargo run --release --bin probe-splash-project -- --max-ticks 3000000`. Compares baseline vs projected memory hash.

### Conformance testing

- Calcite side: `cargo test --workspace`, `probe-splash-project`, `calcite-bench --halt`.
- Chrome conformance: open the emitted CSS in Chrome headless via Playwright (see `tools/fulldiff.mjs` from CSS-DOS side) and read `--memByte_<addr>` values via `getComputedStyle`. **Any PR that changes what Chrome sees must verify at least one small program still renders correctly.**

- [ ] **Step 2: Write the handler body**

In NASM syntax:

```asm
; INT 2F AH=FE — bulk memory op (CSS-DOS vendor extension)
; AL = subfunction (0=fill, 1=memcpy, 2=masked, 3=palette)
; AL=0 (fill): ES:DI = dst, CX = count, DL = byte value
;   Returns: DI advanced by CX, CX=0, CF clear.
int_2f_handler:
    cmp  ah, 0FEh
    jne  .not_ours
    cmp  al, 0
    je   .fill
    ; other subfunctions added in later PRs
    stc                 ; CF=1 = unsupported
    iret
.fill:
    ; Write the bulk-op state into the reserved memory area the
    ; transpiler watches. The transpiler will emit memory-cell CSS
    ; that consumes these locations as --bulkOpKind, --bulkDst,
    ; --bulkCount, --bulkValue.
    ;
    ; One-shot: we don't need to clear these afterwards — they live
    ; at fixed BIOS-reserved addresses and the CSS property
    ; --bulkOpKind reverts to 0 on the next tick via its definition
    ; (see Task 1.2).
    ;
    ; Layout (reserve at 0x500..0x510, well below DOS TPA 0x600):
    ;   0x500: bulkOpKind (1 byte, = 1 for fill)
    ;   0x501: bulkDst lo
    ;   0x502: bulkDst hi
    ;   0x503: bulkCount lo
    ;   0x504: bulkCount hi
    ;   0x505: bulkValue (1 byte)
    ;
    push ax
    push bx
    push es
    push di
    xor  ax, ax
    mov  es, ax         ; ES = 0
    mov  di, 0500h
    mov  [es:di], byte 1   ; kind = 1 (fill)
    mov  ax, di_saved      ; restore original DI from stack
    mov  [es:di+1], al
    mov  [es:di+2], ah
    mov  ax, cx
    mov  [es:di+3], al
    mov  [es:di+4], ah
    mov  [es:di+5], dl
    pop  di
    pop  es
    pop  bx
    pop  ax
    ; The CSS will pick these up during the same tick via the
    ; bulk-op property assignments (Task 1.2).
    mov  cx, 0           ; caller expects CX=0 on success
    ; (ES:DI advance is caller's responsibility — this is a one-shot
    ;  single-tick op; caller already knows dst+count, can do its own
    ;  ADD DI, <count> before/after. Simplest contract: leave DI alone.)
    clc
    iret
.not_ours:
    jmp  prev_int_2f     ; chain to original handler if any
```

Note: the register-save/restore details above need adjustment per the BIOS's calling convention. Read `bios/css-emu-bios.asm`'s existing handlers (e.g. INT 10h, INT 13h) to match the style.

Modify it to prepend a range-predicate clause:

```css

  style(--bulkOpKind: 1) and calc(4096 >= var(--bulkDst)) and calc(4096 < var(--bulkDst) + var(--bulkCount)):
```

- [ ] **Step 3: Remove absorbed memory-cell assignments**

When a bulk-fill binding absorbs a range of memory cells, those cells' normal broadcast-write handling must skip the range-predicate clause — the binding does it natively instead. Approach: track the absorbed property names in a `HashSet<String>` (mirroring `BroadcastResult::absorbed_properties`), and skip them during the normal assignment loop.

Important: the absorbed cells STILL need their normal broadcast-write behaviour (for non-bulk-op ticks). So we don't skip them entirely — we just skip the range-predicate branch. Do this by **not** removing the cells from the assignment list but removing only the range-predicate branch from their expressions before compilation. Alternative simpler design: leave the cells as-is (they still compile correctly, just to inefficient per-cell bytecode) AND run the bulk-fill op first each tick — if it fires, it writes the bytes natively before the per-cell branches run. The per-cell branches then see the already-filled memory and their range-predicate evaluates to the same value (idempotent). This is **correct and simpler**. Prefer it unless profiling shows the per-cell branches are themselves a bottleneck.

- [ ] **Step 4: Execute the bulk fill pre-tick**

Per-tick execution, at the start of `Evaluator::tick`:

```rust
for binding in &self.compiled.bulk_fill_bindings {
    let kind = self.state.read_mem(property_to_address("--bulkOpKind"));
    if kind == 1 {
        let dst = self.state.read_mem(property_to_address(&binding.dst_property)) as i32;
        let count = self.state.read_mem(property_to_address(&binding.count_property)) as i32;
        let val = (self.state.read_mem(property_to_address(&binding.val_property)) & 0xFF) as u8;
        if count > 0 && dst >= 0 {
            let lo = dst as usize;
            let hi = ((dst as i64) + (count as i64)).min(self.state.memory.len() as i64) as usize;
            if hi > lo && lo < self.state.memory.len() {
                self.state.memory[lo..hi].fill(val);
            }
        }
    }
}
```

`property_to_address` is already used in eval.rs — see how it resolves property names to memory addresses.
